CONCEPT 01 · DÉTECTION
IOC — Indicators of Compromise
Indicateurs de Compromission
Un IOC est une preuve concrète qu'une attaque a eu lieu ou est en cours. C'est comme une empreinte digitale laissée par un attaquant. Il peut s'agir d'une adresse IP suspecte, d'un hash de fichier malveillant, d'un nom de domaine utilisé pour de la communication C2 (command & control), ou d'une clé de registre modifiée.
🏠 ANALOGIE DU QUOTIDIEN

Dans une logique d’analogie opérationnelle, une situation comparable consisterait à constater une fenêtre brisée, des traces de boue sur le carrelage et un tiroir ouvert. Sans observation directe de l’auteur, ces traces physiques = des IOC. En cyber, c'est pareil : l'attaquant est peut-être parti, mais il a laissé des empreintes.

🎯 SCÉNARIO SOC RÉEL — Qualification initiale d’un incident

Une alerte SIEM est générée : CONNEXION SORTANTE DÉTECTÉE — 203.0.113.45:4444

Une vérification est effectuée : cette IP est dans la base de données VirusTotal et AbuseIPDB → elle est connue comme serveur C2 du groupe APT28.

C'est un IOC de type IP malveillante. L’indicateur confirme une communication vers une infrastructure malveillante. Une escalade vers le niveau 2 est engagée conformément au processus de traitement.

#️⃣

⚠️ Limite critique des IOC : Un attaquant peut changer son IP en 5 minutes, créer un nouveau domaine, modifier légèrement son malware pour changer son hash. Les IOC sont statiques → faciles à contourner. C'est pour ça qu'on a besoin des TTP (concept 02).

CONCEPT 02 · DÉTECTION
TTP — Tactics, Techniques, Procedures
Le mode opératoire complet d'un attaquant
Les TTP décrivent le COMPORTEMENT d'un attaquant, pas juste une signature. C'est la différence entre "trouver les empreintes" (IOC) et "comprendre comment le cambrioleur opère toujours" (TTP). Tactique = le POURQUOI (l'objectif). Technique = le COMMENT (la méthode). Procédure = le DÉTAIL précis d'exécution.
🕵️ ANALOGIE — Le commissaire de police

Un cambrioleur récidiviste a un mode opératoire : il vise toujours les maisons en angle de rue (Tactique = ciblage), il entre toujours par la fenêtre de la salle de bain au 1er étage (Technique = intrusion), il utilise toujours un tournevis modèle Facom 6mm (Procédure). Même s'il change de ville et d'adresse, son comportement le trahit. C'est la puissance des TTP.

🎯 SCÉNARIO SOC — Analyser un incident en TTP

Tactique : "Persistence" → l'attaquant veut rester dans le système même après redémarrage

Technique : Il crée une clé de registre HKCU\Software\Microsoft\Windows\CurrentVersion\Run avec son malware

Procédure : Il utilise un script PowerShell encodé en base64 pour masquer l'écriture dans le registre

→ Sysmon génère un Event ID 13 (modification registre). Un mapping est réalisé sur MITRE ATT&CK : c'est T1547.001.

CONCEPT 03 · DÉTECTION
IOC vs TTP — La différence fondamentale
Pourquoi les deux sont indispensables
IOC = Statique → C'est une empreinte digitale figée. Facile à générer, facile à partager, mais aussi facile à contourner pour l'attaquant.

TTP = Comportemental → C'est la façon de faire. Beaucoup plus difficile à changer. Un groupe APT garde ses habitudes pendant des années.

🎯 ANALOGIE — Braquer une banque

L'IOC c'est la plaque d'immatriculation de la voiture de fuite → changeable en 10 min. Le TTP c'est le fait qu'il déguise toujours ses complices en agents d'entretien, et attaque toujours le vendredi à 16h → ça, c'est son ADN d'attaquant.

CONCEPT 04 · FRAMEWORK
MITRE ATT&CK
La carte mondiale des attaques réelles
MITRE ATT&CK est une base de données mondiale et gratuite qui répertorie toutes les tactiques et techniques utilisées par les vrais groupes de hackers, documentées à partir d'attaques réelles. C'est la Bible du SOC. Chaque technique a un identifiant unique (ex: T1059.001 = PowerShell).
📚 ANALOGIE — Le Larousse du crime

Imagine un dictionnaire qui recense toutes les techniques de tous les criminels du monde, classées par catégorie, avec des exemples d'utilisation réelle. MITRE ATT&CK c'est exactement ça, mais pour les hackers. Lorsqu’un comportement suspect est observé, ce référentiel permet d’identifier les acteurs susceptibles d’utiliser la technique ainsi que les réponses adaptées.

MITRE ATT&CK — STRUCTURE DES 14 TACTIQUES
# Les 14 étapes qu'un attaquant peut suivre (MITRE Enterprise)

TA0043 → Reconnaissance # Collecter infos sur la cible
TA0042 → Resource Development # Préparer l'infrastructure d'attaque
TA0001 → Initial Access # Entrer dans le réseau (phishing, exploit)
TA0002 → Execution # Exécuter du code malveillant
TA0003 → Persistence # Rester après redémarrage (registre, tâche)
TA0004 → Privilege Escalation # Devenir admin / SYSTEM
TA0005 → Defense Evasion # Contourner antivirus, EDR
TA0006 → Credential Access # Voler mots de passe (Mimikatz)
TA0007 → Discovery # Explorer le réseau interne
TA0008 → Lateral Movement # Se déplacer vers d'autres machines
TA0009 → Collection # Collecter des données sensibles
TA0011 → C2 (Command & Control) # Communiquer avec le serveur hacker
TA0010 → Exfiltration # Envoyer les données volées
TA0040 → Impact # Ransomware, destruction de données
🎯 SCÉNARIO — Mapper un incident sur ATT&CK

Sysmon génère un Event ID 1 : processus cmd.exe lancé par word.exe. C'est suspect.

→ Une consultation de attack.mitre.org permet d’identifier : T1204.002 — User Execution: Malicious File.

→ La tactique est Initial Access / Execution. C'est le début de la kill chain. Une alerte est alors qualifiée : exécution probable d’un document Office malveillant.

Ce mapping ATT&CK te permet de savoir exactement à quel stade se situe l’attaque et quelle est la prochaine étape probable.

CONCEPT 05 · FRAMEWORK
MITRE D3FEND
La réponse défensive à chaque attaque ATT&CK
D3FEND est le pendant défensif de ATT&CK. Pour chaque technique d'attaque documentée dans ATT&CK, D3FEND propose des contre-mesures concrètes. C'est le référentiel de la Blue Team.
🏰 ANALOGIE — L'art de la guerre et la défense

ATT&CK c'est le livre des stratégies d'attaque des envahisseurs. D3FEND c'est le manuel du défenseur qui dit : "Si l'ennemi utilise les échelles (technique d'attaque), alors construis des créneaux et verse de l'huile bouillante (contre-mesure)."

Étape 1 — Reconna. : l'attaquant trouve le nom du DAF sur LinkedIn et son email pro.

Étape 2 — Armement : il crée un fichier Excel avec une macro malveillante.

CONCEPT 07 · PROCESSUS
TLP — Traffic Light Protocol
Qui a le droit de voir cette information ?
Le TLP est un protocole de partage d'information utilisé dans tous les CERT et SOC du monde. Il définit jusqu'où une information peut être diffusée. Tout rapport d’incident ou toute alerte est associé à un niveau TLP de diffusion.
🔴
TLP:RED
Uniquement pour les destinataires nommés. NE PAS partager même en interne.
🟠
TLP:AMBER
Limité à l'organisation et aux partenaires directs. Usage restreint.
🟢
TLP:GREEN
Peut circuler dans la communauté cyber. Pas de diffusion publique.
⬜
TLP:CLEAR
Information publique. Peut être partagée librement, même sur Internet.
🎯 SCÉNARIO — Dans la vraie vie du SOC

L'ANSSI t'envoie un rapport sur un nouveau malware ciblant les hôpitaux. Il est taggué TLP:AMBER.

→ La diffusion interne vers le RSSI et les équipes IT concernées est autorisée.

→ La diffusion publique ou à des tiers non approuvés n’est pas autorisée.

→ Un IOC classifié TLP:RED demeure strictement limité aux destinataires nommés.

CONCEPT 08 · FRAMEWORK INTERNATIONAL
NIST — Cybersecurity Framework
Gestion du risque, gouvernance, incident response et résilience
Le NIST Cybersecurity Framework (CSF) constitue un référentiel international majeur de pilotage du risque cyber. Il structure les capacités de sécurité autour de fonctions cohérentes qui permettent d’aligner gouvernance, sécurité opérationnelle, continuité d’activité et amélioration continue.
🔎
Identify
Inventaire des actifs, compréhension du contexte, cartographie des risques, dépendances critiques.
🛡️
Protect
Contrôles d’accès, durcissement, segmentation, sensibilisation, sécurité des données et des services.
🚨
Detect
Surveillance continue, use cases SIEM, EDR, corrélation, détection d’anomalies et threat intelligence.
⚡
Respond
Gestion d’incident, coordination SOC/CERT, communication de crise, confinement, éradication et décisions.
🔄
Recover
PRA, PCA, restauration, retour à la normale, validation de l’intégrité et amélioration post-incident.
🎯 STRUCTURATION OPÉRATIONNELLE

Le NIST fournit une logique de pilotage lisible aussi bien pour la gouvernance que pour les opérations. Dans un environnement mature, l’inventaire des actifs et des risques relève de Identify, les contrôles préventifs de Protect, la supervision SOC de Detect, la gestion de crise et la réponse à incident de Respond, et les plans de reprise de Recover.

CONCEPT 09 · OUTIL SOC
SIEM — Security Information & Event Management
Plateforme centrale de corrélation et de supervision
Le SIEM est le cœur du SOC. Il collecte, centralise et corrèle des millions de logs provenant de toutes les sources de l'entreprise (Windows, Linux, firewall, proxy, EDR...). Il applique des règles de détection pour générer des alertes quand un comportement suspect est détecté.
🎛️ ANALOGIE — La tour de contrôle de l'aéroport

Les contrôleurs aériens ne regardent pas chaque avion par la fenêtre. Ils ont des écrans qui agrègent les infos de TOUS les avions en même temps (radar, radio, plan de vol). Le SIEM c'est pareil : il agrège tous les "signaux" informatiques de l'entreprise et permet d’identifier l’événement nécessitant une investigation prioritaire.

CONCEPT 10 · OUTIL SOC
SOAR — Security Orchestration, Automation & Response
Automatisation et orchestration de la réponse
Le SOAR automatise les réponses répétitives du SOC. Quand le SIEM génère une alerte, le SOAR peut exécuter automatiquement un playbook (scénario prédéfini) : bloquer une IP, isoler un poste, créer un ticket, envoyer une notification, rechercher l'IOC dans les bases de données... sans intervention humaine.
🤖 ANALOGIE — Le pilote automatique d'avion

Un analyste SOC qui fait les mêmes 20 actions manuelles pour chaque alerte de phishing perd un temps fou. Le SOAR c'est le pilote automatique qui gère les situations connues et simples, afin que l’analyste se concentre sur les turbulences vraiment dangereuses.

🎯 PLAYBOOK SOAR — Réponse automatique à un phishing

Trigger : Un utilisateur signale un email suspect via le bouton "Report Phishing"

Action 1 : SOAR extrait automatiquement les IOC de l'email (URLs, IPs, hash des pièces jointes)

Action 2 : Vérifie chaque IOC sur VirusTotal, AbuseIPDB en 30 secondes

Action 3 : Si malveillant → isole automatiquement le poste de l'utilisateur via EDR

Action 4 : Bloque le domaine malveillant sur le proxy et le DNS

Action 5 : Crée un ticket dans SIRP avec tous les résultats pré-remplis

Action 6 : Transmet à l’analyste SOC un résumé complet

→ Tout ça en moins de 2 minutes, sans intervention humaine.

CONCEPT 11 · OUTIL SOC
SIRP — Security Incident Response Platform
Pilotage structuré des incidents
Le SIRP est la plateforme de gestion des incidents de sécurité. C'est là où chaque incident est documenté, suivi, assigné à un analyste, et résolu. Il assure la traçabilité complète : qui a fait quoi, quand, avec quelle décision. L'outil de référence open source est TheHive.
📋 ANALOGIE — Le logiciel de gestion des urgences à l'hôpital

Quand un patient arrive aux urgences, une fiche est créée : état à l'arrivée, diagnostics, traitements, médecins impliqués. Le SIRP c'est exactement ça pour les incidents cyber : chaque "patient" (incident) a son dossier complet, suivi de A à Z.

CONCEPT 12 · OUTIL SOC
EDR — Endpoint Detection & Response
L'agent de surveillance sur chaque poste
L'EDR surveille en temps réel chaque PC et serveur (= endpoint). Contrairement à l'antivirus classique (qui cherche des signatures connues), l'EDR analyse les comportements : si un processus tente d'accéder à la mémoire de lsass.exe pour voler des mots de passe, l'EDR le détecte même si le malware est inconnu.
📹 ANALOGIE — La caméra de surveillance intelligente

L'antivirus c'est un garde qui compare chaque visiteur à une liste de criminels connus. L'EDR c'est une caméra IA qui analyse les comportements : si quelqu'un essaie d'ouvrir le coffre-fort à 3h du matin en portant des gants, l'alerte se déclenche même si cette personne n'est pas dans la liste des criminels.

CONCEPT 13 · OUTIL SOC
XDR — Extended Detection & Response
L'EDR étendu à tout l'écosystème
Le XDR va au-delà de l'EDR en corrélant les données de TOUS les vecteurs : endpoints, réseau, cloud, email, identité. Il donne une vision unifiée d'une attaque qui traverse plusieurs couches de l'infrastructure.
🎯 SCÉNARIO — Pourquoi l'XDR est supérieur à l'EDR seul

Avec EDR seul : un processus suspect est observé sur HOST-WS03, sans visibilité consolidée sur l’origine et l’étendue de l’attaque.

Avec XDR : la timeline complète devient accessible :

→ Email : phishing reçu et ouvert à 14h22

→ Endpoint : macro exécutée sur HOST-WS03 à 14h24

→ Réseau : connexion vers C2 à 14h26

→ Cloud : tentative d'accès à SharePoint depuis une IP suspecte à 14h30

→ AD : création d'un compte admin fantôme à 14h35

Le XDR t'offre le film complet de l'attaque en un seul écran.

CONCEPT 14 · OUTIL SOC
Sandbox — Bac à sable
Exécuter un virus sans risque
La sandbox est un environnement isolé et jetable dans lequel un fichier suspect peut être exécuté afin d’observer son comportement sans exposition du réseau réel. L’ensemble des actions y est enregistré ce que fait le fichier : connexions réseau, modifications de fichiers, clés de registre créées, processus lancés.
💣 ANALOGIE — Le robot démineur

Les artificiers ne manipulent pas eux-mêmes un engin suspect. Ils envoient un robot dans une zone sécurisée pour analyser la bombe. La sandbox c'est ce robot : elle "ouvre" le malware à ta place, dans une bulle imperméable. Si ça explose, seul le robot virtuel est détruit.

Un email est reçu avec bon_de_commande.pdf.exe. Le fichier est chargé sur any.run (sandbox en ligne gratuite).

En 60 secondes, les éléments suivants sont obtenus :

→ Connexion vers 203.0.113.45:4444 (serveur C2)

→ Création de C:\Users\AppData\Roaming\svchost32.exe

→ Clé registre ajoutée : HKCU\Run\WindowsUpdate (persistance)

→ Communication chiffrée (Cobalt Strike beacon détecté)

→ VERDICT : RAT (Remote Access Trojan). IOCs extraits et partagés en TLP:AMBER.

CONCEPT 16 · RÉSEAU
IDS / IPS
Détection et Prévention d'Intrusion Réseau
IDS (Intrusion Detection System) = surveille le trafic réseau et génère une alerte quand il détecte quelque chose de suspect. Il ne bloque rien.

IPS (Intrusion Prevention System) = fait tout ce que fait l'IDS PLUS il bloque automatiquement le trafic malveillant en temps réel.
🚨 ANALOGIE — Alarme vs Videur

IDS = l'alarme : elle sonne quand quelqu'un entre par effraction, mais elle n'empêche rien. IPS = le videur : il voit quelqu'un de suspect, il l'empêche d'entrer directement.

Risque faux positifFaible (alerte seulement)Élevé (peut bloquer trafic légitime)

CONCEPT 17 · RÉSEAU
Firewall — Pare-feu
Le douanier du réseau
Le firewall filtre le trafic réseau selon des règles : autoriser ou bloquer selon la source, la destination, le port et le protocole. C'est la première barrière de défense périmétrique.
🛂 ANALOGIE — Le douanier à la frontière

Il vérifie l'origine, la destination et le type de "marchandise". Il laisse passer le trafic HTTPS (port 443) depuis Internet, mais bloque le port RDP (3389) depuis l'extérieur. Il ne regarde pas LE CONTENU des valises (ça c'est le WAF ou l'IPS), juste QUI va OÙ avec QUOI.

RÈGLES FIREWALL TYPIQUES (pfSense / FortiGate)
# Règles de base d'un SOC
ALLOW ANY → INTERNET TCP 443 (HTTPS)
ALLOW ANY → INTERNET TCP 80 (HTTP)
BLOCK INTERNET → ANY TCP 3389 (RDP) # No RDP externe
BLOCK INTERNET → ANY TCP 22 (SSH) # No SSH externe
ALERT ANY → C2_IPLIST ANY # IP malveillantes connues
BLOCK ANY → C2_IPLIST ANY # Bloque C2 connus

⚠️ SNORT vs SURICATA — La question d'entretien classique

Snort = monofil (single-threaded), plus simple, appartient maintenant à Cisco, règles Snort Community gratuites. Parfait pour apprendre.

Suricata = multifil (multi-threaded), plus performant sur gros trafic, analyse TLS, génère des logs JSON (compatible ELK), supporte les mêmes règles Snort. Préféré en production aujourd'hui.
SURICATA — RÈGLE DE DÉTECTION (format Snort/Suricata)
# Règle Suricata : détecter un scan Nmap
alert tcp any any -> $HOME_NET any (
msg:"Nmap SYN Scan détecté";
flags:S;
threshold: type both, track by_src, count 20, seconds 1;
classtype:network-scan;
sid:1000001; rev:1;
);

# → Si 20 SYN sans ACK en 1 seconde depuis la même IP = scan de ports
→ Suricata génère une alerte et l'envoie dans ELK/Wazuh

msg:"Téléchargement EXE depuis HTTP non chiffré";
http.uri; content:".exe"; endswith;

);

CONCEPT 40 · MAÎTRISE DES LOGS
Maîtrise des logs, corrélation et visibilité sécurité
Journalisation, normalisation, corrélation, rétention et exploitation SOC
La visibilité cyber dépend directement de la qualité des logs. Une organisation mature définit quelles sources journaliser, quels événements conserver, combien de temps les retenir, comment les centraliser et comment les exploiter dans une logique d’alerte, d’enquête et de preuve.

🎯 LIEN DIRECT AVEC LE SOC ET LE PENTEST

Le SOC doit savoir où journaliser, quels signaux surveiller et quels comportements sont normaux ou anormaux selon la plateforme. Un pentest autorisé ne s’aborde pas de la même façon sur un serveur Linux, un contrôleur de domaine Windows, une application web, un tenant Azure ou un compte AWS. La maîtrise de ces environnements montre une capacité de lecture transversale très recherchée en cybersécurité.

⚠️ Point de maturité : la cybersécurité ne se limite pas à un outil unique. Elle suppose la compréhension combinée des systèmes, des applications, des identités, des réseaux, du cloud, des journaux et des dépendances de l’organisation.

🎯 LECTURE ORGANISATIONNELLE

Une organisation mature articule les rôles entre SOC, gouvernance, architecture et métiers. Le RSSI pilote la stratégie, le SOC surveille et détecte, les équipes techniques implémentent et les experts interviennent en cas d’incident ou d’audit.

⚠️ Vision clé : comprendre les métiers permet de se positionner, d’évoluer et de dialoguer efficacement avec chaque acteur de la cybersécurité.
